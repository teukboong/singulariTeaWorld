# Host Worker Contract

`singulari-world` keeps the simulator standalone. It owns world storage,
pending-turn records, visual job records, and redacted prompts. The embedding
host owns model session dispatch and image generation.

The host worker is the process boundary between those two systems.

## Dispatch Backends

Text turn dispatch has three backend modes:

- `host-session-api`
  - Intended main path.
  - The embedding app injects a bounded worldsim event into the active agent
    session through an official host API.
  - The reference CLI does not reverse-engineer private app endpoints. Until an
    official host API exists, this backend emits `host_session_turn_required`
    and stops there.

- `codex-exec-resume`
  - Reference fallback path.
  - Uses `codex exec resume <thread-id> -` after a world is bound with
    `codex-thread-bind` or `--codex-thread-id`.
  - Dispatch records prevent duplicate turns for the same world/turn/thread.

- `manual`
  - Development path.
  - Emits the exact pending turn and command hints without dispatching.

Image generation has one supported path:

- host image generation
  - The worker emits `codex_app_image_generate_required` after claiming a visual
    job.
  - The embedding host calls its image-generation capability and saves a PNG to
    the returned `destination_path`.
  - The host completes or releases the job through the CLI or MCP.

There is no `codex exec` image fallback.

## Reference CLI

Start a one-shot host worker:

```bash
singulari-world host-worker \
  --world-id <world-id> \
  --once \
  --text-backend manual \
  --no-visual-jobs
```

Run the current text fallback:

```bash
singulari-world codex-thread-bind \
  --world-id <world-id> \
  --thread-id <codex-thread-id> \
  --codex-bin "$(command -v codex)"

singulari-world host-worker \
  --world-id <world-id> \
  --text-backend codex-exec-resume \
  --interval-ms 750
```

Run with host-owned image jobs:

```bash
singulari-world host-worker \
  --world-id <world-id> \
  --text-backend codex-exec-resume \
  --claim-visual-jobs \
  --interval-ms 750
```

`--claim-visual-jobs` claims at most one unclaimed visual job per tick and emits
the claim payload. The embedding host must either save the PNG and complete the
job, or release the claim on failure.

## JSONL Events

Every host worker event is one JSON object per line with schema version
`singulari.host_worker_event.v1`.

Startup:

```json
{"event":"worker_started","world_id":"stw_example","text_backend":"codex-exec-resume"}
```

Main text path waiting for the embedding app:

```json
{"event":"host_session_turn_required","world_id":"stw_example","turn_id":"turn_0001","host_capability":"codex_app.thread.dispatch"}
```

Fallback text dispatch:

```json
{"event":"codex_exec_dispatch_started","world_id":"stw_example","turn_id":"turn_0001","pid":12345}
```

Manual text path:

```json
{"event":"manual_agent_turn_required","world_id":"stw_example","turn_id":"turn_0001"}
```

Image job waiting for a host claim:

```json
{"event":"visual_job_available","world_id":"stw_example","slot":"stage_background"}
```

Claimed image job requiring host generation:

```json
{"event":"codex_app_image_generate_required","world_id":"stw_example","slot":"stage_background","claim_id":"..."}
```

Idle:

```json
{"event":"worker_idle","world_id":"stw_example"}
```

## Failure Rules

- Hidden adjudication context never appears in host worker image prompts or VN
  packets.
- `host-session-api` fails closed in the reference CLI. It only emits a host
  action event.
- `codex-exec-resume` requires a bound thread. Without one, it emits
  `codex_exec_dispatch_blocked`.
- Visual jobs are never generated by the simulator binary. Claimed jobs must be
  completed or released by the embedding host.
- Dispatch records and visual claims are durable world-store files, so worker
  restarts do not duplicate already-started jobs.
